# Guidelines
The skill should be able to deal with different design themes. So in the future, with a new design theme, the skill should be able to create a presentation with the new design theme without any problems.

# Architecture

1. The skill is described in skill.md file. It contains the description of the skill.
2. We are following a slide and card appraoch.
    a. Slides have a layout in regrads to rows and columns. There are different types of layouts available. Each layout has a different number of rows and columns. Slides have a title, footer, footerline, headerline, logo, background and content. The content is made up of cards.
    b. Cards have a type. The type defines the content of the card. For example, a text card has a text content, an image card has an image content. You can find the different cards in the C:\localworkspace\agent-skills\skills\presentation-design\cards folder. Cards follow a predefined structure with title, titleline, subtitle, body, footer and so on. You can investigate cards to find out more about the structure of the cards. This approach should not be touched for the future, but we need to ensure, that these elements can be displayed in a more professional and attractive way. This will make the presentations more attractive and interesting for the audience.
    c. We have a basic design theme which is defined in C:\localworkspace\agent-skills\skills\presentation-design\themes\base.css. When we scaffold a new presentation, we create a new css-file which is called theme.css. Every design token which is defined in the theme.css overrides the design token which is defined in the base.css. So if we want to change the design of the presentation, we can simply change the design tokens in the theme.css file. This makes it easy to create new design themes in the future.
    d. we can also use the design tokens in the presentation-definition.md file and override styles for specific elements. This is done by using "style_overrides" in the presentation-definition.md file. This allows us to change the design of specific elements in the presentation without changing the overall design theme. It should be possible to show/supress/modify specific elements in the presentation by using the "style_overrides" in the presentation-definition.md file. For example, if we do not want to show the title of a card, we can simply set "show_title: false" in the style_overrides of the card in the presentation-definition.md file. This will override the default setting of the card which is defined in the theme.css file. This allows us to create more flexible and customizable presentations in the future.
    e. cards and their purpose are defined in yaml-files. the yaml-files should act as a registry for cards. when an agent want to create a presentation, it can look into the yaml-files to find out which cards are available and what is the purpose of each card. This will make it easier for the agent to create a presentation with the right cards and the right content. The yaml-files should also contain information about the design of the cards. For example, if we have a text card, we can define in the yaml-file that the text should be displayed in a certain font size, color and so on. This will make it easier for the agent to create a presentation with a consistent design. The yaml-files should be stored in a central location where all agents can access them. This will ensure that all agents have access to the same information about the cards and their design.
